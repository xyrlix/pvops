## 一、原始方案核心提炼

### 1.1 智子熹源 V3.2 在卖什么

| 维度 | 内容 |
| --- | --- |
| 核心产品 | 光伏运维专家智能体（知识库 + 大模型 + 智能体协同） |
| 落地形态 | 管理层/分析层/执行层三层视角，从场站 → 逆变器 → 组串逐层下钻 |
| 价值指标 | 管理效率提升 25 倍+、等效发电小时数提升 5%~10%、综合收益提升 5%~8% |
| 技术卖点 | AI 诊断根因 + 证据链、报告自动生成、工单联动、反馈记忆增强 |
| 商业模式 | 公有云 / 私有化 / 混合云订阅 |

### 1.2 对方未覆盖、我们可以切入的机会点

1. **储能闭环**：仅做光伏运维，未涉及储能 EMS 与充放电策略优化。
2. **电力交易**：未涉及现货市场、虚拟电厂（VPP）聚合与报量报价。
3. **效果量化**：价值指标偏口号，缺少“每单诊断/每笔策略的收益可核算”。
4. **私有化深度**：电站业主对数据出域敏感，私有化部署 + 本地模型是强卖点。
5. **开放集成**：与大企业已有的 ERP/MES/财务系统打通能力弱。

## 二、双视角评审：原 MD 计划书的优势与风险

### 2.1 系统架构师视角

#### ✅ 做得好的地方

- **六层架构清晰**：采集 → 存储 → 分析 → 业务服务 → AI 智能体 → 前端，职责边界合理。
- **技术选型主流**：Next.js + FastAPI + TDengine + PostgreSQL + LangChain，工程化风险低。
- **预留了扩展性**：储能、交易阶段作为二期、三期，不破坏一期架构。
- **数据链路闭环**：采集 → 指标 → 异常 → 诊断 → 工单，符合运维场景真实闭环。

#### ⚠️ 需要补强的风险点

| 风险点 | 说明 | 建议 |
| --- | --- | --- |
| **协议碎片化被低估** | 逆变器品牌多、寄存器地址差异大，做 3 个品牌就想覆盖 MVP 可能不够 | 抽象“设备模型 + 协议驱动”两层，先以华为/阳光/固德威为主，其他通过配置化驱动扩展 |
| **AI 诊断准确率** | 大模型直接做根因分析容易“一本正经胡说八道” | 必须“规则引擎兜底 + LLM 解释增强 + 人工反馈校正”三件套，首批只出“建议”不自动执行 |
| **时序数据量级** | 1 个 10MW 电站，若 1 分钟级采集，一年轻松过亿条 | TDengine 选型合理，但需提前设计降采样、冷热分离、 retention policy |
| **边缘网关可靠性** | 站端网络不稳定，采集程序必须本地缓存 + 断点续传 | 边缘侧用 SQLite/本地文件做缓冲，上报失败自动重试 |
| **储能/交易接入的合规与安全** | 下发充放电策略、报量报价属于控制类指令，影响电网和资产安全 | 控制指令必须“建议 → 人工确认 → 双因子/白名单下发”，并留审计日志 |
| **多租户与数据隔离** | 私有化卖给多个业主时，集团/场站/角色权限复杂 | 早期就引入 RBAC + 数据行级隔离，避免后期大改 |

### 2.2 业务专家视角

#### ✅ 做得好的地方

- **对标明确**：1:1 复刻对方 UI 与核心能力，降低产品定义风险。
- **差异化路径合理**：光伏切入 → 储能 → 交易，符合新能源资产运营演进趋势。
- **商业模式清晰**：私有化买断 + 年服务费 + 效果分润，贴合国内业主付费习惯。

#### ⚠️ 需要补强的风险点

| 风险点 | 说明 | 建议 |
| --- | --- | --- |
| **MVP 范围过大** | 6 个月想做“采集 + 大屏 + AI 诊断 + 储能 + 交易”，对最小团队偏紧 | 建议把 MVP 严格限定为“光伏运维智能体”可演示，储能作为 1.5 阶段，交易作为独立二期 |
| **价值验证滞后** | 到 W12 才验收 MVP，中间缺少可售卖的中间形态 | 每 2 周出一个可演示的“客户可见价值点”，如 W4 能看实时数据，W8 能出 AI 诊断报告 |
| **客户成功体系缺失** | 私有化部署后需要持续运营、知识库维护、模型调优 | 计划中需增加“客户成功/远程运维”角色或 SLA |
| **报价模型未细化** | 效果分润听起来好，但“提升发电量 5%”如何归因、如何对账，是商务地雷 | 必须在合同中明确基准线、计量方式、第三方校核机制 |
| **电力交易资质门槛** | 售电公司牌照、VPP 聚合资质、电网接口审批周期长 | 交易阶段建议先做“模拟申报 + 策略建议”，真正入市作为三期后半段 |

## 三、产品定位与差异化边界

### 3.1 一句话定位

> **“面向新能源资产运营商的 AI 运营决策平台”**：先帮电站看清风险、挽回损失，再帮储能套利增收，最终参与电力市场交易，让资产越用越值钱。

### 3.2 三阶段产品边界（修正版）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  阶段一（M1-M3）        阶段二（M4-M6）        阶段三（M7-M12+）              │
│  光伏运维智能体           储能运营增强包          电力交易 / VPP 套件           │
│  ───────────────         ───────────────         ────────────────             │
│  · 多源数据采集           · BMS/PCS 接入          · 现货价格预测              │
│  · 指标/健康度/异常         · 发电/负荷预测         · 报量报价策略             │
│  · 三层下钻大屏            · 峰谷套利/需量管理       · 虚拟电厂聚合             │
│  · AI 诊断 + 证据链         · 策略下发 + 收益复盘     · 需求响应 + 结算对账        │
│  · 工单闭环               ·                    · 绿电/绿证/碳资产             │
│  ★ 可独立售卖 MVP ★       ★ 高毛利模块 ★         ★ 平台级壁垒 ★              │
└─────────────────────────────────────────────────────────────────────────────┘
```

> **关键修正**：原 MD 把三阶段都压在 6 个月内，节奏过紧。建议把**光伏运维 MVP 放宽到 3 个月**，确保质量；储能 3 个月；交易作为 6 个月后的持续演进。若你坚持 6 个月全量，则需增加 1~2 名核心开发或大幅缩减范围。

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  数据感知层  │ → │  认知分析层  │ → │  决策推荐层  │ → │  执行闭环层  │
├─────────────┤    ├─────────────┤    ├─────────────┤    ├─────────────┤
│ 逆变器/组串  │    │ 健康度评分   │    │ AI 诊断建议  │    │ 工单派发    │
│ 气象/电表   │    │ 异常识别    │    │ 充放电策略   │    │ 策略确认下发 │
│ BMS/PCS    │    │ 损失量化    │    │ 交易申报方案 │    │ 效果复盘    │
│ 市场价格    │    │ 基线对比    │    │ 风险预警    │    │ 案例沉淀    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### 4.2 核心业务域

| 业务域 | 核心职责 | 关键产出 |
| --- | --- | --- |
| **资产管理域** | 场站、设备、组串、BMS/PCS 等资产台账与拓扑关系 | 资产图谱、设备档案 |
| **数据采集域** | 协议适配、边缘采集、数据治理、消息总线 | 干净、对齐的时序数据 |

| **储能运营域** | SOC 监控、发电/负荷/电价预测、优化求解、策略下发 | 每日套利收益报告 |
| **电力交易域** | 市场数据接入、报量报价、VPP 聚合、结算对账 | 交易收益与风险报告 |

## 五、系统架构设计（分层 + 关键数据流）

### 5.1 修正后的六层架构

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ L6  交互层     │ Next.js 14 + Tailwind + ECharts + AG Grid                  │
│               │ 大屏 / 工作台 / AI Copilot / 移动端 H5 / 开放 API 门户       │
├──────────────────────────────────────────────────────────────────────────────┤
│ L5  AI 智能体层 │ LangChain / LangGraph + RAG + 多 Agent 编排                │
│               │ 诊断 Agent / 报告 Agent / 工单 Agent / 问答 Agent           │
│               │ (二期) 调度 Agent / (三期) 交易 Agent                        │
├──────────────────────────────────────────────────────────────────────────────┤
│ L4  业务服务层  │ FastAPI 微服务 + 领域驱动设计（DDD）                         │
│               │ 资产 / 告警 / 工单 / 诊断 / 报告 / 策略 / 交易 / 权限 / 计费   │
├──────────────────────────────────────────────────────────────────────────────┤
│ L3  分析计算层  │ 指标引擎 / 规则引擎 / 健康度模型 / 预测模型 / 优化求解器      │
│               │ 流式计算（实时）+ 批处理（日/月 rollup）                      │
├──────────────────────────────────────────────────────────────────────────────┤
│ L2  数据存储层  │ TDengine（时序）+ PostgreSQL（业务）+ Redis（缓存/会话）     │
│               │ + PGVector/Milvus（向量）+ MinIO（对象）+ SQLite（边缘缓存） │
├──────────────────────────────────────────────────────────────────────────────┤
│ L1  边缘采集层  │ 边缘网关 + 协议驱动（Modbus / IEC104 / MQTT / 厂家 API）     │
│               │ 本地缓存、断点续传、安全上报、远程 OTA                       │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 5.2 关键数据流

#### 数据流 A：实时采集 → 入库 → 指标计算

```
设备(Modbus/MQTT) → 边缘网关 → EMQX → Kafka → 流处理(Flink/自定义) → TDengine
                                        ↓
                                   PostgreSQL(元数据/告警)
```

> **简化建议**：MVP 期 Kafka 可用 Celery + Redis 替代，降低运维复杂度；数据量上来后再引入 Kafka/Flink。

#### 数据流 B：异常 → AI 诊断 → 工单 → 反馈

```
规则引擎命中异常 → 告警去重聚合 → 诊断 Agent（数据校验/根因/建议）
                                              ↓
                                报告 Agent / 工单 Agent → 工单系统 → 现场执行
                                              ↑
                                         人工反馈 → 案例库 → RAG/Prompt 增强
```

#### 数据流 C：储能策略生成与下发（二期）

```
BMS/PCS 数据 + 气象/NWP + 负荷数据 + 电价数据
              ↓
      预测模型（发电/负荷/电价）
              ↓
      优化求解器（MILP / 启发式）
              ↓
      策略建议 → 人工/自动确认 → EMS/PCS 下发
              ↓
      实际执行对比 → 收益复盘 → 模型迭代
```

> **安全底线**：所有控制类下发必须支持“模拟模式”，正式上线初期强制人工确认，并保留完整审计链。

## 六、核心领域模型

### 6.1 资产模型

```text
集团(Tenant)
  └── 场站(Station)
       ├── 设备(Device)
       │    ├── 逆变器(Inverter)
       │    ├── 汇流箱(Combiner)
       │    ├── 电表(Meter)
       │    ├── 气象站(WeatherStation)
       │    └── 储能单元(ESSUnit)  [二期]
       └── 组串(String)
            └── 组件(Module)  [可选]
```

### 6.2 告警/异常模型

| 字段 | 说明 |
| --- | --- |
| `alert_id` | 告警唯一 ID |
| `source` | 规则引擎 / AI 诊断 / 人工上报 |
| `severity` | 紧急 / 重要 / 一般 / 提示 |
| `device_id` | 关联设备 |
| `metric` | 触发指标 |
| `root_cause` | 根因（AI 生成或规则命中） |
| `evidence` | 证据链（截图、基线、规则） |
| `estimated_loss_kwh` | 预估损失电量 |
| `estimated_loss_cny` | 预估损失金额 |
| `action_guide` | 修复建议 |
| `work_order_id` | 关联工单 |
| `feedback` | 人工反馈（正确/部分正确/错误） |

### 6.3 收益核算模型（关键差异化）

每个 AI 建议或策略都附带可核算的收益：

```text
建议收益 = 挽回发电量(kWh) × 电价(元/kWh) - 执行成本(元)
储能套利收益 = 放电收益 - 充电成本 - 寿命损耗成本 - 需量罚款减免
交易收益 = 市场出清收益 - 偏差考核 - 机会成本
```

> 这是向对方“效果可量化”口号落地的核心设计。

## 八、技术选型复核

## 九、商业化与定价策略

### 9.1 目标客户画像

| 客户类型 | 痛点 | 付费能力 | 推荐切入点 |
| --- | --- | --- | --- |
| 中小型光伏电站业主（1~10MW） | 运维人手少、故障发现慢 | 中等 | 标准化 SaaS 或轻量私有化 |
| 大型电站运营商 / 国企能源公司 | 场站多、数据分散、需合规 | 强 | 私有化部署 + 定制开发 |
| 工商业储能 + 光伏一体用户 | 想套利增收 | 强 | 储能运营增强包 |
| 售电公司 / VPP 运营商 | 需聚合资源参与市场 | 强 | 电力交易套件 |

### 9.2 收费模式

| 模式 | 说明 | 适用阶段 |
| --- | --- | --- |
| **私有化买断** | 一次性软件授权费 + 部署费 | MVP 期主力 |
| **年度服务费** | 维护、升级、模型调优、远程运维 | 持续现金流 |
| **效果分润** | 按挽回损失/套利收益的 5%~15% 分成 | 需有可信的第三方计量基准 |
| **SaaS 订阅** | 按场站数/接入容量月费 | 中小客户，后期规模化 |

### 9.3 定价参考（需根据客户规模调整）

- **光伏运维智能体私有化版**：15~50 万/场站（含部署）
- **储能运营增强包**：10~30 万/站 + 年服务费 3~8 万
- **电力交易套件**：按接入容量或交易额抽成

## 总结

原 MD 计划书已经是一份合格的“需求开发计划书”，但作为系统计划书，它需要：

1. **更严格的分阶段边界**：MVP 只聚焦光伏运维，储能/交易不混入一期。
2. **更工程化的架构设计**：补充协议驱动层、边缘缓存、控制安全、多租户隔离。
3. **更业务化的收益模型**：把“提升 5%”的口号，落地为每个 Action 的可核算收益。
4. **更务实的排期**：9 个月稳健版优于 6 个月赶工版。
