## 一、四视角技术选型评审

当前我们锁定的技术栈是：

| 层级 | 选型 |
| --- | --- |
| 前端 PC/H5 | Next.js 14 / 建议改为 Vue 3 + Vite |
| 小程序 | Uni-app |
| 后端 | FastAPI (Python) |
| 时序库 | TDengine |
| 业务库 | PostgreSQL |
| 缓存/队列 | Redis + Celery |
| MQTT | Mosquitto |
| 部署 | Docker Compose + Cloudflare Tunnel |
| AI | LangChain + RAG + DeepSeek/通义千问 |

### 1.1 业务架构师视角

#### 1.1.1 选型是否支撑商业模式？

| 业务需求 | 技术支撑 | 评估 |
| --- | --- | --- |
| 快速交付 MVP | Python + Vue/React 全栈开发效率高 | ✅ 支撑 |
| 私有化部署 | Docker Compose 一键部署 | ✅ 支撑 |
| 按场站/容量授权 | PostgreSQL 多租户 + RBAC | ✅ 支撑 |
| 后续储能/交易扩展 | FastAPI 模块化解耦，算法库丰富 | ✅ 支撑 |
| 客户希望尽早体验 | 2 周一个可演示迭代 | ✅ 支撑 |
| 效果分润商业模式 | 需要精确的指标计算和审计日志 | ⚠️ 需重点设计 |

#### 1.1.2 成本与风险

- **开发成本**：Python + Vue 人才市场供应充足，AI/算法工程师也主要用 Python。
- **运维成本**：Docker Compose 运维简单，不需要专职 DevOps。
- **替换风险**：TDengine、PostgreSQL、Redis 都是主流开源产品，无供应商锁定。
- **扩展风险**：Cloudflare Tunnel 免费版够用，但大客户可能要求 VPN/专线，需保留切换空间。

**业务结论**：技术选型与商业模式匹配，成本可控，无重大商业风险。

### 1.2 安全专家视角

#### 1.2.1 当前架构的安全优势

| 设计 | 安全价值 |
| --- | --- |
| Cloudflare Tunnel | 无需公网 IP，不暴露内网端口，只出向 443 |
| FastAPI + Pydantic | 强类型校验，天然防御大量注入和类型攻击 |
| PostgreSQL + SQLAlchemy | ORM 防止 SQL 注入 |
| Docker 隔离 | 服务间网络隔离，单点被攻破影响有限 |
| 私有化部署 | 数据不出客户域，降低合规风险 |

#### 1.2.2 必须补齐的安全措施

| 风险点 | 等级 | 对策 |
| --- | --- | --- |
| 传输加密 | 🔴 高 | Cloudflare Tunnel 已提供 TLS，内部容器间通信若跨机需 mTLS |
| 身份认证 | 🔴 高 | JWT + Refresh Token，密码 Argon2 哈希，支持 MFA（二期） |
| 权限控制 | 🔴 高 | RBAC + 数据行级隔离（集团/场站/角色） |
| API 限流 | 🟡 中 | 防暴力破解和 DDoS，按用户/IP 限流 |
| 敏感配置泄露 | 🔴 高 | 密码/Token 走 Docker Secrets 或 .env，禁止硬编码 |
| 控制指令安全 | 🔴 高 | 储能/交易下发必须人工确认 + 审计日志 + 操作回滚 |
| 数据备份 | 🟡 中 | 自动备份 TDengine + PostgreSQL，异地容灾 |
| 日志审计 | 🟡 中 | 记录登录、操作、控制指令、异常访问 |
| 依赖安全 | 🟡 中 | 定期扫描 Python/Node 依赖漏洞（safety/dependabot） |
| 边缘网关安全 | 🟡 中 | 网关只出向连接，不开放入站端口，固件 OTA 签名 |

#### 1.2.3 安全红线

1. **任何控制类指令（储能充放电、交易申报）默认必须人工确认**，不可自动执行。
2. **生产环境禁止使用 `root` 账号运行容器**。
3. **所有 API 必须鉴权**，包括内部服务间调用。
4. **敏感操作必须记录审计日志**，保留至少 180 天。

**安全结论**：基础架构安全偏上，但必须系统性地补齐认证、授权、审计、备份四块。

### 1.3 系统架构师视角

#### 1.3.1 当前架构的优点

- **简单**：6 个容器即可跑起来，维护成本低。
- **解耦**：时序/业务/缓存分离，职责清晰。
- **可扩展**：FastAPI 模块化，未来可按业务域拆分。
- **IoT 友好**：Python + Mosquitto + Modbus 生态成熟。
- **AI 友好**：LangChain + RAG + Python 算法库无缝衔接。

#### 1.3.2 当前架构的潜在短板

| 短板 | 说明 | 建议 |
| --- | --- | --- |
| 单点故障 | 单容器运行，无高可用 | MVP 期可接受，大客户阶段加主从/集群 |
| 无服务发现 | 容器名直接通信 | Docker Compose 内够用，拆微服务后再引入 |
| 日志分散 | 各容器独立日志 | 增加 Loki 或集中式日志脚本 |
| 监控缺失 | 无应用性能监控 | 增加 Prometheus + Grafana（二期） |
| 配置管理简单 | .env 文件管理 | 客户多时用配置中心或模板化 |

#### 1.3.3 是否需要微服务？

**MVP 期不需要**。

微服务解决的问题（独立部署、独立扩展、团队并行）在 MVP 期都是伪需求。当前单体 FastAPI 足够支撑：
- 几十个场站
- 几百台设备
- 几十并发用户

**何时拆分**：
- 场站数 > 500 或日活用户 > 1000
- 不同业务域需要独立发布节奏
- 团队超过 8 人，需要并行开发

**系统架构结论**：当前架构对 MVP 和早期商业化“够用且优秀”，无需为扩展而扩展。

## 二、前端技术选型深度分析

### 2.1 前端一定要用 Node.js 吗？

**不是“必须”，但“现代前端工程化离不开 Node.js 工具链”**。

无论你选 React、Vue 还是 Angular，构建打包都需要 Node.js。但 Node.js 只在**开发构建阶段**使用，生产环境可以输出纯静态文件由后端 serve。

如果你非常排斥 Node.js，唯一替代方案是：
- 纯 HTML + JavaScript + ECharts
- 或 HTMX + 后端渲染

但这只适合非常简单页面，**不适合复杂的数据大屏和交互式工作台**。所以结论是：Node.js 作为构建工具保留，生产环境不需要 Node 运行时。

### 2.2 Vue vs React/Next.js 怎么选？

| 维度 | Vue 3 + Vite | Next.js 14 (React) |
| --- | --- | --- |
| 学习曲线 | 平缓，中文文档友好 | 较陡，概念多 |
| 开发效率 | 高，模板语法直观 | 高，但配置复杂 |
| 大屏生态 | ECharts Vue 封装成熟 | ECharts React 封装成熟 |
| 表格组件 | Element Plus / Arco Design / Vuetify | AG Grid / TanStack Table |
| 小程序生态 | Uni-app 就是 Vue 系，天然一致 | Taro 支持 React，但生态略弱 |
| 团队招聘 | 国内 Vue 开发者多 | React 开发者也多 |
| SSR/SEO | Nuxt.js 可补 | Next.js 原生强 |
| 打包体积 | Vite 轻量 | Next.js 较重 |

### 2.3 针对你的项目，我推荐 Vue 3 + Vite

理由：

1. **小程序一致性**：你计划做小程序，Uni-app 是 Vue 语法。如果 PC/H5 也用 Vue，团队可以一套技术栈打天下。
2. **维护简单**：Vue 3 + Vite 项目比 Next.js 更轻量，配置少，出问题容易定位。
3. **国内生态友好**：Element Plus、Arco Design、Ant Design Vue 等中后台组件库成熟，适合运维后台类系统。
4. **不需要 SSR**：光伏运维系统是内网/私有化应用，不需要 SEO，Next.js 的 SSR 优势用不上。
5. **大屏场景 Vue 完全胜任**：ECharts 有 `vue-echarts`，AG Grid 也有 Vue 版本。

### 2.4 推荐前端技术栈

```
PC Web / H5：Vue 3 + Vite + TypeScript + Pinia + Vue Router
UI 组件库：Element Plus 或 Arco Design
图表：ECharts（vue-echarts）
表格：AG Grid Vue / Vxe-Table
小程序：Uni-app（Vue 3）
APP：Uni-app 打包 或 Flutter（后期）
```

### 2.5 前端是否需要 SSR？

**不需要**。

- 这是内网/私有化系统，不需要 SEO。
- 数据都是异步加载，首屏慢一点可以接受。
- SSR 增加部署复杂度，与“维护简单”原则冲突。

## 三、测试与质量保障体系

### 3.1 核心原则

1. **没有真实设备也要能测试**：用设备模拟器生成数据。
2. **测试左移**：在开发阶段就写单元测试，不是最后补。
3. **自动化优先**：CI 自动跑测试，不通过不能合并。
4. **分层测试**：单元 → 集成 → E2E，逐级覆盖。

### 3.2 后端测试（Python）

#### 3.2.1 单元测试

```python

    assert abs(pr - 0.833) < 0.001
```

## 四、开发流程与协作规范

### 4.1 开发模式：敏捷迭代

- **2 周一个 Sprint**
- **每周一个小版本**（可内部演示）
- **每 2 周一个客户可见版本**（MVP 演示）
- **每日 15 分钟站会**（远程可异步）

### 4.2 Git 工作流

推荐 **Trunk-based Development**（主干开发）或轻量 Git Flow：

```
main        ← 生产分支，随时可部署
  ↑
develop     ← 开发分支，集成测试通过
  ↑
feature/*   ← 功能分支，生命周期 < 3 天
  ↑
hotfix/*    ← 紧急修复
```

规范：
- 功能分支从 `develop` 切出
- 通过 PR 合并，必须 1 人 Review
- 合并前 CI 通过
- 提交信息规范：`feat:`, `fix:`, `docs:`, `test:`, `refactor:`

### 4.3 环境管理

| 环境 | 用途 | 数据 |
| --- | --- | --- |
| **本地 dev** | 开发调试 | 模拟数据 |
| **测试环境** | 自动化测试/内部演示 | 模拟数据 + 少量真实数据 |
| **Staging** | 客户预览 | 真实数据（脱敏） |
| **生产环境** | 客户正式使用 | 真实数据 |

### 4.4 文档规范

- **API 文档**：FastAPI 自动生成 `/docs`（OpenAPI/Swagger）
- **架构文档**：每次重大变更更新 Markdown
- **部署文档**：《私有化部署手册》
- **运维文档**：《常见问题排查手册》
- **CHANGELOG**：每个版本记录变更

### 4.5 代码规范

**Python**：
- Ruff 格式化
- 类型注解
- Pydantic 模型校验
- 异常统一处理

**Vue/TypeScript**：
- ESLint + Prettier
- Composition API
- 组件单文件 < 400 行
- API 调用统一封装到 `services/`

### 5.4 MVP 演示重点

每次给客户演示，都要讲清楚：**“你看到了什么、解决了什么问题、能产生什么价值”**。

例如 M3 演示话术：
> “这是您 5 个场站的实时状态。这个红色气泡是 XX 场站，过去 7 天损失了 3200 度电，按 0.8 元/度算，损失了 2560 元。点击进去可以看到是 3 号逆变器组串离散率异常，AI 判断可能是遮挡，建议本周清洗。”

## 六、迭代更新与持续交付机制

### 6.1 迭代节奏

```
每周：内部小版本（bug 修复、小功能）
每 2 周：Sprint 版本（客户可见功能）
每月：发布说明 + 客户培训
每季度：大版本规划回顾
```

### 6.2 客户反馈闭环

```
客户使用 → 收集反馈 → 优先级排序 → 进入 Sprint
    ↑                                    ↓
  发布更新 ← 开发测试 ← 代码实现 ← 需求确认
```

**反馈收集渠道**：
- 系统内置反馈按钮
- 客户成功经理定期回访
- 每次演示后的讨论

### 6.3 灰度发布策略

- 先在测试环境验证
- 再选一个友好客户做 Beta 测试
- 最后全量推送
- 关键版本保留回滚能力（Docker 镜像标签）

### 6.4 远程更新流程

```bash
# 客户服务器上的升级脚本 update.sh
#!/bin/bash
set -e
cd /opt/pvops
git pull origin main
docker compose pull
docker compose down
docker compose up -d
# 数据库迁移
docker compose exec backend alembic upgrade head
# 健康检查
curl -f http://localhost:8000/api/v1/health
echo "Update completed"
```

## 七、总结与下一步

### 7.1 四视角结论

| 视角 | 结论 |
| --- | --- |
| **业务架构师** | 技术选型与商业模式匹配，成本可控，建议 Vue 统一小程序/Web |
| **安全专家** | 基础架构安全偏上，但必须补齐认证、授权、审计、备份 |
| **系统架构师** | 当前架构对 MVP 够用且优秀，无需过早微服务化 |
| **测试专家** | 测试体系需从零建立，设备模拟器是 MVP 期关键 |

### 7.2 前端最终建议

**Vue 3 + Vite + Uni-app**。

- 与小程序技术栈一致
- 维护简单
- 国内生态友好
- 完全满足大屏和移动端需求

### 7.3 测试与质量关键动作

1. 建立设备数据模拟器
2. 后端 pytest 单元测试覆盖核心算法
3. 前端 Vitest + Playwright 测试
4. CI 流水线强制跑 lint/test/build
5. 安全扫描纳入 CI

### 7.4 下一步建议

如果你认可这个分析，建议按以下顺序推进：

1. **确认前端技术栈**：Vue 3 是否接受？
2. **输出项目骨架**：Vue + FastAPI + Docker Compose + 测试框架
3. **建立设备模拟器**：没有真实设备也能开发
4. **跑通第一个 Sprint**：数据采集 + 基础大屏，2 周后内部演示
