# 光伏运维智能体 · 简化技术架构方案 V4

> **设计原则**：维护简单、可控、稳定够用、不过度设计  
> **部署方式**：Docker Compose + Caddy + Cloudflare Tunnel  
> **目标**：1~2 人小团队能长期运维，不需要专职 DevOps  
> **编制日期**：2026-06-23

---

## 一、架构调整核心思路

你提到要用 **Caddy** 和 **Cloudflare Tunnel**，这是一个非常务实的选择。基于此，我把整体架构做以下简化：

| 原方案 | 简化方案 | 理由 |
| --- | --- | --- |
| Nginx / Kong API 网关 | **Caddy** | 配置更简单，自动 HTTPS，反向代理够用 |
| 公网 IP + 端口暴露 | **Cloudflare Tunnel** | 不需要公网 IP 和防火墙开墙，远程访问更安全 |
| 微服务拆分 | **单体 FastAPI** | MVP 期业务不复杂，单体开发快、部署简单 |
| AI 单独服务 | **AI 作为后端内部模块** | 减少一个容器，降低调用复杂度 |
| EMQX + Kafka | **Mosquitto + Redis + Celery** | 轻量 MQTT broker，Redis 同时做缓存/队列 |
| K8s 后期规划 | **Docker Compose 长期够用** | 大客户真要上 K8s 也是一年以后的事 |

## 三、关键组件说明

### 3.1 Caddy（反向代理 + 自动 HTTPS）

**为什么选 Caddy？**

- 配置极其简单，一个 `Caddyfile` 搞定
- 自动申请和续期 HTTPS 证书（Let's Encrypt/ZeroSSL）
- 自带反向代理、负载均衡、静态文件服务
- 日志清晰，排查问题方便

**典型 Caddyfile**：

```caddyfile
{
    auto_https off  # 内网或 Cloudflare Tunnel 场景关闭自动 HTTPS
}

# 内部域名或 localhost
pvops.local {
    # 前端静态文件
    handle_path / {
        root * /srv/web
        file_server
        try_files {path} /index.html
    }

    # 后端 API
    handle_path /api/* {
        reverse_proxy backend:8000
    }

    # WebSocket 实时推送
    handle_path /ws/* {
        reverse_proxy backend:8000
    }
}
```

> 如果走 Cloudflare Tunnel，Caddy 主要做内部服务路由，HTTPS 由 Cloudflare 边缘处理。

---

### 3.2 Cloudflare Tunnel（远程访问）

**为什么选 Cloudflare Tunnel？**

- 不需要公网 IP
- 不需要在客户防火墙开墙（只出向 443）
- 免费版够用
- 每个客户可以分配独立子域名，如 `station-a.pvops.example.com`

**典型部署方式**：

```bash
# 在客户服务器安装 cloudflared
docker run -d --name cloudflared \
  cloudflare/cloudflared:latest tunnel --no-autoupdate run --token <TUNNEL_TOKEN>
```

**流量路径**：

```
用户浏览器/手机 → Cloudflare 边缘 → Cloudflare Tunnel → 客户内网 → Caddy → 后端/前端
```

> **注意**：Cloudflare Tunnel 适合远程运维和中小客户访问。如果客户是大型国企且有严格网络安全策略，可能需要改为 VPN 或专线。但 90% 的私有化场景 Tunnel 都够用。

### 3.3 后端：单体 FastAPI

**为什么不做微服务？**

- MVP 期业务域之间耦合高，拆服务增加通信复杂度
- 单体部署只需要一个容器，升级简单
- 一个人能看懂全部代码

**内部模块划分**（同一个代码库，按目录组织）：

```
/backend
├── app/
│   ├── api/              # API 路由
│   │   ├── stations.py   # 场站
│   │   ├── devices.py    # 设备
│   │   ├── alarms.py     # 告警
│   │   ├── work_orders.py # 工单
│   │   ├── diagnosis.py  # AI 诊断
│   │   ├── reports.py    # 报告
│   │   └── strategy.py   # 策略（二期）
│   ├── services/         # 业务逻辑
│   ├── models/           # 数据模型
│   ├── agents/           # AI Agent 模块
│   ├── tasks/            # Celery 异步任务
│   ├── core/             # 配置、数据库连接、缓存
│   └── main.py           # FastAPI 入口
```

**Celery Worker**：同一个代码库，单独启动 worker 容器即可：

```bash
# API 服务
uvicorn app.main:app --host 0.0.0.0 --port 8000

# 异步任务 worker
celery -A app.tasks worker --loglevel=info

# 定时任务 scheduler（可选）
celery -A app.tasks beat --loglevel=info
```

### 3.4 前端：Next.js 静态导出

**简化方案**：

- 开发时用 Next.js 完整模式
- 部署时用 `next export` 导出静态文件
- Caddy 直接 serve 静态文件

**优点**：
- 不需要 Node.js 运行时
- 部署简单，就是一堆 HTML/JS/CSS
- Caddy 处理静态文件性能很好

**限制**：
- 不能直接用 SSR/ISR
- 但光伏运维大屏主要是客户端渲染（ECharts），静态导出完全够用

**如果某些页面需要 SSR**，可以保留一个 Next.js Node 容器，但大多数场景静态导出即可。

---

### 3.5 数据存储简化

| 组件 | 用途 | 是否必需 |
| --- | --- | --- |
| **TDengine** | 时序数据 | ✅ 必需 |
| **PostgreSQL** | 业务数据 | ✅ 必需 |
| **Redis** | 缓存 + 任务队列 + 会话 | ✅ 必需 |
| **MinIO** | 文件存储 | ⚠️ 可选 |
| **本地目录** | 小规模存报告/图片 | ✅ 可用 |

**MinIO 可以省掉吗？**

- 如果客户只有 1~5 个场站，报告和图片直接存本地目录或 PostgreSQL 大字段即可
- 等客户规模大了再上 MinIO

---

### 3.6 消息与采集简化

| 组件 | 原方案 | 简化方案 |
| --- | --- | --- |
| MQTT Broker | EMQX | **Mosquitto**（轻量，Docker 一键启动） |
| 消息队列 | Kafka | **Redis + Celery** |
| 边缘网关 | 独立复杂程序 | **Python 脚本 + 本地 SQLite 缓存** |

**为什么 Mosquitto 够用了？**

- MVP 期设备数量少（几十到几百台）
- Mosquitto 单机可支持万级连接
- 配置简单，资源占用低

**为什么不用 Kafka？**

- Kafka 运维重，需要 ZooKeeper/KRaft
- Redis Streams 或 Celery 足够处理 MVP 数据量
- 等每天数据量过亿条再上 Kafka

## 四、简化后的 Docker Compose

```yaml
version: "3.8"

services:
  caddy:
    image: caddy:2-alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - ./web/dist:/srv/web
      - caddy_data:/data
      - caddy_config:/config
    depends_on:
      - backend

  web:
    build: ./frontend
    # 开发模式用 next dev，生产模式只负责 build 出静态文件给 Caddy
    command: sh -c "npm run build && cp -r dist /output"
    volumes:
      - ./web/dist:/output

  backend:
    build: ./backend
    command: uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
    environment:
      - DATABASE_URL=postgresql://user:pass@postgres:5432/pvops
      - TDENGINE_URL=taos://root:taosdata@tdengine:6030/pvops
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      - postgres
      - tdengine
      - redis

  worker:
    build: ./backend
    command: celery -A app.tasks worker --loglevel=info
    environment:
      - DATABASE_URL=postgresql://user:pass@postgres:5432/pvops
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      - redis
      - postgres

  scheduler:
    build: ./backend
    command: celery -A app.tasks beat --loglevel=info
    depends_on:
      - redis

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: pvops
    volumes:
      - postgres_data:/var/lib/postgresql/data

  tdengine:
    image: tdengine/tdengine:3.3.0.0
    volumes:
      - tdengine_data:/var/lib/taos

  redis:
    image: redis:7-alpine
    volumes:
      - redis_data:/data

  mosquitto:
    image: eclipse-mosquitto:2
    ports:
      - "1883:1883"
    volumes:
      - ./mosquitto.conf:/mosquitto/config/mosquitto.conf

  cloudflared:
    image: cloudflare/cloudflared:latest
    command: tunnel --no-autoupdate run --token ${CF_TUNNEL_TOKEN}
    restart: unless-stopped

volumes:
  postgres_data:
  tdengine_data:
  redis_data:
  caddy_data:
  caddy_config:
```

> **说明**：
> - `web` 服务只在构建静态文件时运行，生产环境由 Caddy 直接 serve
> - `worker` 和 `scheduler` 可以合并成一个容器，也可以分开
> - `cloudflared` 可选，客户有公网 IP 时可去掉

## 五、远程运维与升级策略

### 5.1 如何远程维护客户部署？

| 方式 | 工具 | 场景 |
| --- | --- | --- |
| Web 管理后台 | 系统自带 | 日常配置、查看数据 |
| SSH 隧道 | Cloudflare Tunnel 或 Tailscale | 紧急登录服务器 |
| 日志查看 | Caddy 日志 + Docker logs | 排查问题 |
| 版本升级 | Docker 镜像更新 + docker compose up -d | 发版 |

### 5.2 升级流程

```bash
# 1. 客户服务器拉取新版本
cd /opt/pvops
git pull

# 2. 重新构建并启动
docker compose down
docker compose build
docker compose up -d

# 3. 数据库迁移（如有）
docker compose exec backend alembic upgrade head
```

> 建议把升级脚本写成 `update.sh`，客户或运维人员一键执行。

## 六、与上一版架构的对比

| 方面 | V3 原方案 | V4 简化方案 |
| --- | --- | --- |
| 部署复杂度 | 中等（Nginx + 可能 K8s） | 低（Docker Compose + Caddy） |
| 远程访问 | 公网 IP 或 VPN | Cloudflare Tunnel，无需公网 IP |
| 运维人员要求 | 需要一定 DevOps 经验 | 普通开发/运维可操作 |
| 容器数量 | 10+ | 7~9 |
| 服务拆分 | 业务 + AI 分离 | 合并为单体 FastAPI |
| 消息队列 | EMQX + Kafka | Mosquitto + Redis/Celery |
| 扩展性 | 好 | 够用，后期可拆分 |

## 七、仍然保留的核心设计

虽然做了简化，但以下设计不会妥协：

1. **API-first**：所有客户端共用同一套 API，未来扩展方便。
2. **数据库分离**：时序数据走 TDengine，业务数据走 PostgreSQL，不混用。
3. **边缘缓存**：采集网关本地 SQLite 缓存，断网可续传。
4. **AI 可解释**：规则兜底 + LLM 解释，不盲目依赖大模型。
5. **控制安全**：储能/交易控制类指令默认建议模式，人工确认后才下发。

## 八、总结

这套简化架构的核心是：**用 Caddy 替换 Nginx，用 Cloudflare Tunnel 解决远程访问，用单体 FastAPI 替换微服务，用 Mosquitto + Redis 替换 EMQX + Kafka**。

结果是：
- 部署只要 `docker compose up -d`
- 配置只要一个 `Caddyfile`
- 远程访问只要一个 `cloudflared` token
- 1~2 人小团队可以完全掌控
